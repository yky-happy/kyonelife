# 动态菜单 + 动态路由方案，对本项目是否值得做？

> 决策时间：2026-05-22
> 背景：登录联调已通，下一步要决定走"动态菜单 + 动态路由"，还是跳过它直接做业务 CRUD

---

## 一、这套方案具体在做什么

### 后端要做的

1. 新建 `MenuController`，至少两个接口：
   - `GET /admin/menu/tree` —— 根据当前登录管理员的角色，查询 `role_menu` 关联表，递归组装出菜单树（含 `CATALOG` 和 `MENU` 类型节点），返回类似：
     ```json
     [
       { "id": 1, "title": "内容管理", "type": "CATALOG",
         "children": [
           { "id": 10, "title": "文章管理", "path": "/content/article", "component": "content/article/index" }
         ]
       }
     ]
     ```
   - `GET /admin/menu/perms` —— 返回当前用户的按钮权限标识列表（如 `["article:add", "article:delete"]`），供前端按钮级权限控制
2. 加上 Sa-Token 的注解检查，确保不同角色返回不同菜单

### 前端要做的

1. 登录成功后，立刻发请求拉菜单树
2. 把树结构里的 `component` 字符串映射成 `() => import('@/views/.../index.vue')`，调用 `router.addRoute()` **动态注册路由**
3. Sidebar 从写死的菜单数组改成读 Pinia 里的菜单树
4. 用一个 `v-permission` 自定义指令，控制按钮显示（基于 perms 接口返回的标识）
5. 处理"刷新页面后路由丢失"的问题（动态路由不持久，刷新就没了，需要在路由守卫里判断 + 重新拉）

**保守估计开发量**：后端 0.5 天，前端 1 天（动态路由 + 刷新保持的坑较多），共 **1.5 天**。

---

## 二、这套方案在企业项目里为什么是"标配"

在多人多角色的企业后台，动态菜单解决的真问题是：

1. **不同角色看到的菜单不同**：财务、运营、技术 Lead 看到的左侧菜单完全不一样，写死在前端就没法做
2. **菜单变动不需要前端发版**：新增一个"运营报表"页面，运维在数据库里加一行就能上线，前端不用重新打包部署
3. **按钮级别权限**：同一个页面里，普通员工只能查看，主管能编辑，财务能审批 —— 必须靠后端权限标识控制
4. **可审计**：所有权限分配走数据库，谁有什么权限一目了然，符合企业合规要求

这是个**真实业务诉求**驱动的方案，所以企业项目里几乎人人都做。

## 三、回到本项目：这些诉求你有几个？

诚实对照一下：

| 诉求 | 你的实际情况 | 是否需要 |
|------|--------------|----------|
| 不同角色看到不同菜单 | 你**就是唯一管理员**，没有其他角色登录管理后台 | ❌ |
| 菜单变动不需要前端发版 | 个人博客菜单几乎不变，每次改完都是你自己重新部署 | ❌ |
| 按钮级权限控制 | 单管理员场景，所有按钮你都能用 | ❌ |
| 合规审计 | 个人项目，无此要求 | ❌ |
| 简历技术亮点 | 这个**确实有价值** | ✅ |
| 学习 RBAC 完整流程 | 想系统学一遍权限设计 | ✅（可选）|

四个**真实业务诉求全部用不到**，剩下的是简历亮点和学习价值——这两个值不值 1.5 天投入，是你要权衡的。

## 四、不做的话，你失去什么 / 得到什么

### 不做（保持现状：Sidebar 写死，路由静态注册）

**失去**
- 简历上少一行"基于 RBAC 的菜单 + 路由动态加载"
- 后端 `menu` / `role` / `role_menu` 三张表（你已经设计好了的）成了摆设
- 错过一次实操"前后端配合做权限系统"的机会

**得到**
- **省 1.5 天**，可以提早开始阶段 3、4（标签、合集、文章 CRUD），让博客**真正长出内容**
- 路由配置直观，所有页面在 `router/index.ts` 一眼看完
- 避免动态路由特有的坑：刷新页面 404、面包屑取不到、Vue Router 4 的类型推断等

### 做（按原计划推进阶段 2）

**失去**
- 1.5 天时间，期间博客没有任何用户可见的进展
- 这段时间消耗你最大的精力，可能影响后续阶段的连续性

**得到**
- 一份"看起来很专业"的权限系统
- 简历项目里可以写完整的 RBAC 章节
- 数据库表设计**没白费**

## 五、给你的判断标准

回答下面几个问题，直接对应该做不该做：

1. **这个博客是给自己用，还是想当作完整作品集去面试展示？**
   - 给自己用 → 不做
   - 作品集展示 → 考虑做

2. **你简历里已经有"RBAC / 权限系统"相关项目了吗？**
   - 有 → 不做（重复点没意义）
   - 没有，且想往后端/全栈方向找工作 → 做

3. **你对 Vue Router 动态路由、按钮级权限指令、菜单树渲染这套**实际感兴趣吗？**
   - 感兴趣，想学透 → 做（学习收益是真的）
   - 只是按计划机械执行 → 不做（很容易做到一半放弃）

4. **你打算这个项目什么时候上线 / 完成？**
   - 一周内 → **绝对不做**
   - 一个月内 → 不做，重心放在内容
   - 半年慢慢打磨 → 可以做

## 六、推荐路线

### 强烈推荐：**先跳过，后面回头补**

理由：

1. **现在跳过没有任何沉没成本**。你后端的 `menu` / `role` / `role_menu` 表设计已经完成，等于已经"占好了坑"。前端 Sidebar 当前是写死的菜单数组，未来要换成动态菜单，**只是替换 Sidebar 的数据源**，已写好的页面不用动。
2. **博客的核心价值是"文章发布"**，不是后台架构。先把阶段 4（文章 CRUD + Markdown 编辑器）做完，让博客**真的能写东西**，上线后能看到内容。
3. **RBAC 在简历里要有内容支撑才有说服力**。如果你简历写"实现了 RBAC 动态菜单"，但博客只有 2 篇文章，面试官会觉得"项目本身价值不大，只是练了一下技术"。反过来，如果博客有 30 篇高质量技术文章 + 完整的标签合集体系，面试官会觉得"这是个活的项目"。
4. **未来要加进去成本极低**。等阶段 3、4 走完，再回头花 1 天加动态菜单，那时候你对项目结构更熟，做起来更快。

### 折中方案：**只做按钮权限，不做菜单动态化**

如果你最在意"按钮级权限"这个技术点，可以走轻量路线：

- Sidebar 继续写死（因为你就一个管理员，菜单不会变）
- 但是后端登录返回的 `LoginVO` 里多加一个 `perms: List<String>` 字段
- 前端写一个 `v-permission="'article:delete'"` 自定义指令，根据 perms 控制按钮显隐

这个改动**只需要 2~3 小时**，简历上同样能写"实现了细粒度的按钮级权限控制"，但避开了动态路由的所有复杂性。

## 七、最终建议

| 你的状态 | 建议 |
|---------|------|
| 想尽快上线、内容优先 | ✅ **跳过**，直奔阶段 3 |

| 想要简历亮点但不想折腾太久 | ✅ 走**折中方案**（只做按钮权限） |
