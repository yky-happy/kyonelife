# 2026.06.23 数据聚合与 Redis 缓存开发记录

## 一、今日完成的功能

今天主要围绕“数据驱动能力”和“Redis 性能优化”做了三组功能：

1. 行为埋点数据聚合
2. 数据看板增强
3. Redis 缓存与实时统计接入

整体链路从原来的：

```text
前台埋点 -> MySQL event_log 明细表 -> 后台直接查询明细
```

升级为：

```text
前台埋点
-> MySQL event_log 明细表
-> 定时聚合到每日统计表
-> 后台数据看板查询聚合表
-> Redis 缓存热点查询与实时统计
```

这样可以减少后台看板直接扫描明细表的压力，也让项目具备更完整的数据分析链路。

### 2.2 聚合任务实现

新增了聚合服务：

```text
AnalyticsAggregationService
AnalyticsAggregationServiceImpl
```

核心逻辑：

```text
每 5 分钟执行一次
-> 读取当天 event_log 明细
-> 按日期、事件类型、文章、标签、合集、关键词聚合
-> 使用 ON DUPLICATE KEY UPDATE 幂等写入统计表
```

幂等设计的原因：

定时任务可能重复执行，如果直接累加会导致统计数据越来越大。现在采用“按当天明细重新计算，然后覆盖更新”的方式，重复执行不会造成重复计数。

```yaml
      password: ***REMOVED***
```

另外，`blogRedis` 原配置中存在两条 `bind`，后面的：

```text
bind 127.0.0.1 -::1
```

会导致 Redis 只监听容器内部地址，宿主机 Spring 无法连接。已将其注释掉，保留：

```text
bind 0.0.0.0
```

并重启 `blogRedis` 生效。

### 5.2 热点数据缓存

新增通用 Redis JSON 缓存服务：

```text
RedisCacheService
RedisKeys
```

已接入缓存的接口：

| 数据 | 缓存时间 |
| --- | --- |
| 前台文章分页 | 60 秒 |
| 文章归档 | 3 分钟 |
| 标签列表 | 3 分钟 |
| 合集列表 | 3 分钟 |
| 数据看板接口 | 30 秒 |

实现方式：

```text
先查 Redis
-> 命中则反序列化返回
-> 未命中则查 MySQL
-> 将结果序列化为 JSON 写入 Redis，并设置 TTL
```

Redis 不可用时会降级为直接查 MySQL，不影响正常访问。

### 5.3 文章阅读量 Redis 缓存 + 定时刷库

原逻辑：

```text
访问文章详情 -> 直接 UPDATE article.view_count
```

新逻辑：

```text
访问文章详情
-> Redis Hash 自增阅读量增量
-> 返回 MySQL 基础阅读量 + Redis 增量
-> 每 2 分钟定时批量刷回 MySQL
```

使用的 Redis Key：

```text
kyonelife:article:view-delta
```

定时任务：

```text
ArticleViewCountServiceImpl.flushToDb()
```

这样可以避免高频访问文章详情时频繁写 MySQL。

如果 Redis 不可用，会降级为：

```text
直接原子更新 MySQL view_count
```

保证文章详情页不会因为 Redis 故障而不可用。

### 5.4 埋点实时统计

新增实时统计服务：

```text
RealtimeStatsService
RealtimeStatsServiceImpl
```

当前 Redis 实时统计包括：

| 指标 | Redis 数据结构 |
| --- | --- |
| 当日 PV | String + INCR |
| 当日 UV | HyperLogLog |
| 实时热门文章 | ZSet |

使用的 Redis Key 示例：

```text
kyonelife:realtime:pv:20260623
kyonelife:realtime:uv:20260623
kyonelife:realtime:hot-articles
```

实时统计目前作为 Redis 层能力保留，后台看板主要仍以 MySQL 聚合表为准。后续可以继续扩展为“实时看板”。
