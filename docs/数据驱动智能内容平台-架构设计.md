### 2.2 前端 SDK 设计

```javascript
// 极简埋点 SDK（blog-web/src/utils/tracker.js）
class Tracker {
  constructor(options) {
    this.endpoint = options.endpoint   // 上报接口地址
    this.userId = options.userId       // 当前用户 ID（未登录为 null）
    this.sessionId = this._getSessionId()
    this.queue = []                    // 本地缓冲队列
    this._initAutoTrack()
    this._startBatchUpload()           // 每 5s 批量上报
  }

  track(eventType, properties = {}) {
    this.queue.push({
      eventType,
      userId: this.userId,
      sessionId: this.sessionId,
      deviceType: this._getDeviceType(),
      properties,
      timestamp: Date.now()
    })
  }

  _startBatchUpload() {
    setInterval(() => {
      if (this.queue.length === 0) return
      const batch = this.queue.splice(0, 20)    // 每批最多 20 条
      navigator.sendBeacon(this.endpoint, JSON.stringify(batch))
    }, 5000)
  }
}
```

**关键设计点**：
- 使用 `navigator.sendBeacon` 而非 `axios`，页面关闭时也能可靠上报
- 本地缓冲队列 + 批量上报，减少请求次数
- 未登录用户用 sessionId（localStorage UUID）标识

### 2.3 后端接入接口

```java
// 埋点接入接口（高并发写，不做复杂业务逻辑）
@PostMapping("/api/track/event")
public Result<Void> trackEvent(@RequestBody List<TrackEventDTO> events,
                               HttpServletRequest request) {
    // 1. 补充服务端字段
    String ip = IpUtil.getClientIp(request);
    events.forEach(e -> {
        e.setIp(ip);
        e.setReceiveTime(System.currentTimeMillis());
    });
    // 2. 直接发到 MQ，不做同步处理
    trackEventProducer.sendBatch(events);
    return Result.ok();
}
```

**设计原则**：接入接口只做两件事：补充服务端字段 + 发 MQ，不查库、不做业务，保证高吞吐。

## 三、数据管道层（消息队列）

### 3.1 为什么需要 MQ

| 问题 | 没有 MQ | 有 MQ |
|------|---------|-------|
| 流量突增（文章爆火） | 直接写库，数据库压力暴增 | MQ 削峰，消费者按固定速率消费 |
| ClickHouse 短暂不可用 | 数据丢失 | 消息积压在 MQ，恢复后继续消费 |
| 埋点逻辑变更 | 需要修改接入接口 | 只需新增/修改消费者 |
| 多个下游（存储+分析+告警） | 接入接口耦合所有逻辑 | 多个消费者独立订阅 |

### 3.2 技术选型

**推荐：RabbitMQ**（个人博客场景）

理由：
- Spring Boot 有 `spring-boot-starter-amqp`，集成简单
- 消息确认机制成熟，不丢数据
- 单机部署，运维简单
- 日均百万级消息完全够用

如果想展示更强的技术深度（投大厂）：**RocketMQ**，支持事务消息、消息回溯。

## 四、数据存储层（ClickHouse）

### 4.1 为什么用 ClickHouse 而不是 MySQL

| 查询场景 | MySQL | ClickHouse |
|---------|-------|-----------|
| 某篇文章过去 30 天每日 PV | 全表扫描，慢 | 列存储，秒级 |
| 某用户最近 100 次行为 | 快（主键查询） | 同样快 |
| 全站所有文章的平均阅读深度 | 极慢 | 毫秒级聚合 |
| 标签点击热力分布 | 慢 | 快 |

**核心差异**：ClickHouse 是列式存储，聚合查询性能比 MySQL 快 100x+，专为分析场景设计。

### 4.2 表结构设计

```sql
-- 用户行为事件表（ClickHouse）
CREATE TABLE user_events (
    event_id     String,
    event_type   LowCardinality(String),   -- 枚举类型，节省存储
    user_id      UInt64,
    session_id   String,
    article_id   UInt64,
    tag_id       UInt64,
    category_id  UInt64,
    ip           String,
    device_type  LowCardinality(String),
    read_depth   UInt8,                    -- 0-100
    time_spent   UInt32,                   -- 秒
    event_time   DateTime,
    receive_time DateTime
) ENGINE = MergeTree()
  PARTITION BY toYYYYMM(event_time)        -- 按月分区，方便过期清理
  ORDER BY (event_type, user_id, event_time)
  TTL event_time + INTERVAL 1 YEAR;       -- 1年后自动删除
```

### 4.3 典型分析查询

```sql
-- 文章阅读量趋势（过去 30 天，按天统计）
SELECT
    toDate(event_time) AS date,
    count() AS pv,
    uniq(user_id) AS uv
FROM user_events
WHERE event_type = 'article_read'
  AND article_id = 42
  AND event_time >= now() - INTERVAL 30 DAY
GROUP BY date
ORDER BY date;

-- 用户兴趣标签分布（用于构建用户画像）
SELECT
    tag_id,
    count() AS click_count,
    sum(time_spent) AS total_time
FROM user_events
WHERE user_id = 1001
  AND event_type IN ('tag_click', 'article_read')
  AND event_time >= now() - INTERVAL 90 DAY
GROUP BY tag_id
ORDER BY total_time DESC
LIMIT 20;

-- 全站热门文章排行（实时）
SELECT
    article_id,
    count() AS pv,
    uniq(user_id) AS uv,
    avg(read_depth) AS avg_depth
FROM user_events
WHERE event_type = 'article_read'
  AND event_time >= now() - INTERVAL 24 HOUR
GROUP BY article_id
ORDER BY pv DESC
LIMIT 10;
```

## 六、智能推荐模块

### 6.1 推荐流程

```
用户请求推荐
     │
     ▼
┌─────────────┐
│  召回阶段    │  从候选集（全量文章）中筛出几百篇
├─────────────┤
│ 协同过滤召回 │  → 相似用户喜欢的文章
│ 标签匹配召回 │  → 与用户兴趣标签重合的文章
│ 热门召回     │  → 过去 24h 热门文章（保底）
└──────┬──────┘
       │
       ▼
┌─────────────┐
│  排序阶段    │  对召回结果打分排序
├─────────────┤
│ 标签匹配度   │  用户画像 × 文章标签权重
│ 时效性       │  发布时间越近分越高
│ 质量分       │  点赞数 / 评论数 / 平均阅读深度
│ 多样性       │  同分类文章打压，避免信息茧房
└──────┬──────┘
       │
       ▼
┌─────────────┐
│  过滤阶段    │  去掉已读文章、屏蔽用户、违规内容
└──────┬──────┘
       │
       ▼
   返回 Top 10
```

### 6.2 核心排序公式

```
score = tagMatchScore × 0.4
      + recencyScore × 0.2
      + qualityScore × 0.3
      + diversityBonus × 0.1

其中：
tagMatchScore = Σ(用户标签权重 × 文章标签权重) / 归一化系数
recencyScore  = e^(-λ × 发布天数)          // 指数衰减
qualityScore  = log(1 + 点赞数) × avgDepth / 100
```

### 6.3 LLM 增强排序（可选）

@EventListener
public void onArticlePublished(ArticlePublishedEvent event) {
    Article article = event.getArticle();

    // 1. 文章分块（每块约 500 字，保留上下文重叠）
    List<String> chunks = documentSplitter.split(article.getContentMd());

            .articleId(article.getId())
            .chunkIndex(i)
            .content(chunks.get(i))
            .embedding(vectors.get(i))
            .build());
    }
}

## 十二、面试高频问题准备

**Q：ClickHouse 和 MySQL 的核心区别是什么？你为什么选 ClickHouse？**
> MySQL 是行式存储，适合 OLTP（高频读写单行）；ClickHouse 是列式存储，适合 OLAP（大量数据聚合分析）。埋点场景是典型的写多读少、以聚合查询为主，ClickHouse 的 MergeTree 引擎在这类查询上比 MySQL 快 100 倍以上。

**Q：MQ 消费失败了怎么办？会不会丢数据？**
> 消费者采用手动 ACK 模式，只有成功写入 ClickHouse 后才确认消息。消费失败会触发重试（最多 3 次），超过重试次数进入死信队列，告警后人工处理。RabbitMQ 本身开启持久化，即使宕机消息也不丢失。

**Q：用户画像是实时的还是离线的？**
> 两者结合。增量更新：每次用户行为事件消费时，直接 incr Redis 中的标签权重，实时性强但可能有误差。全量重算：每天凌晨用 ClickHouse 历史数据全量重算，覆盖 Redis，保证准确性。

**Q：RAG 的检索效果不好怎么优化？**
> 几个方向：①分块策略优化，从固定长度分块改为按语义段落分块；②Embedding 模型升级（小模型换大模型）；③混合检索，结合关键词匹配（BM25）和向量检索的结果重排；④引入 Reranker 模型对 Top-K 结果做精排。

**Q：推荐系统如何避免信息茧房？**
> 在排序公式里加入多样性惩罚：同一分类的文章累计曝光超过阈值后，该分类后续文章降权；同时保留 10-20% 的随机探索流量，推送用户未接触过的分类文章。
