# kyonelife 博客项目去同质化最终改进方向

## 一、核心判断

普通博客项目在秋招里确实非常同质化。

如果项目只停留在：

```text
文章 CRUD
标签 CRUD
评论 CRUD
登录注册
```

那么竞争力有限。后续不能继续只堆普通功能，而要把 `kyonelife` 改造成：

```text
内容管理平台
+ 对象存储
+ 行为埋点
+ 数据分析看板
+ AI 内容辅助
+ 工程治理能力
```

最终对外表达不再是“个人博客”，而是：

```text
一个支持内容生产、内容展示、用户行为分析和 AI 辅助创作的个人内容平台。
```

## 二、最终项目定位

建议将项目定位为：

```text
数据驱动的智能内容管理与展示平台
```

这个定位比“个人博客系统”更适合秋招展示。

它包含三条主线：

```text
内容生产：后台写文章、上传图片、AI 摘要、AI 标签推荐
内容消费：前台文章展示、搜索、标签、合集、归档
数据反馈：行为埋点、PV/UV、阅读时长、热门文章、数据看板
```

## 三、推荐最终亮点组合

优先级从高到低：

```text
1. MinIO 文件上传
2. 操作日志系统
3. 运行日志分析系统
4. 行为埋点系统
5. 数据分析看板
6. AI 摘要生成
7. AI 标签推荐
8. 搜索系统
```

其中真正能拉开差距的是：

```text
行为埋点 + 数据看板 + AI 内容辅助
```

但前提是先把 MinIO 和基础工程能力补齐。

## 四、为什么要区分三类日志

项目里容易把“日志”混在一起，但真实业务系统会区分不同用途。

建议将日志能力拆成三类：

```text
操作日志：记录管理员做了什么
运行日志：分析系统运行得怎么样
埋点日志：统计用户看了什么、点了什么
```

三者定位不同：

```text
操作日志系统：后台管理审计，用于追踪管理员登录、发布文章、修改标签、删除数据等行为
运行日志分析系统：系统运行监控，用于分析后端启动关闭、接口耗时、WARN/ERROR、慢请求和异常接口
行为埋点系统：用户行为分析，用于统计 PV/UV、热门文章、阅读时长、标签点击和搜索关键词
```

这样项目表达会更清晰：

```text
操作日志体现后台审计能力
运行日志分析体现运维监控能力
行为埋点体现数据分析能力
```

## 五、运行日志分析系统设计

### 目标

基于后端已有的日志文件，提供轻量级运行状态分析能力。

当前项目已经有日志文件：

```text
blog/logs/kyonelife-blog.log
```

日志中包含：

```text
后端启动时间
后端关闭时间
数据库连接池启动和关闭
HTTP 请求路径
HTTP 状态码
接口耗时
访问 IP
WARN / ERROR 信息
业务异常信息
```

### 核心功能

第一版支持：

```text
查看最近 200 行日志
按 INFO / WARN / ERROR 筛选
按关键词搜索日志
统计今日请求总数
统计今日 WARN / ERROR 数量
统计慢请求列表
统计接口访问排行
统计接口平均耗时
展示后端最近启动和关闭时间
```

### 可分析的问题

运行日志分析可以回答：

```text
今天后端是否正常启动
今天后端是否正常关闭
有没有 ERROR
有没有 WARN
哪个接口最慢
哪个接口访问最多
哪些接口出现了业务异常
是否存在未登录访问
是否存在文章不存在等异常访问
```

### 接口设计

建议新增后台接口：

```text
GET /admin/log/recent
GET /admin/log/search?keyword=文章不存在
GET /admin/log/summary
GET /admin/log/slow-requests
GET /admin/log/top-apis
```

### 返回示例

```json
{
  "todayRequestCount": 184,
  "warnCount": 6,
  "errorCount": 0,
  "slowRequestCount": 1,
  "lastStartTime": "2026-06-02 14:08:03",
  "lastShutdownTime": "2026-06-02 15:15:00"
}
```

### 后台页面

建议新增：

```text
系统日志页
运行状态概览
慢请求分析
异常日志列表
接口访问排行
```

### MVP 实现

第一版不需要引入 ELK，也不需要复杂日志平台。

建议：

```text
后端读取本地 log 文件
用正则解析 HTTP 请求日志
用正则解析 WARN / ERROR 日志
按日期过滤当天记录
在内存中聚合统计
后台页面展示分析结果
```

可以先只解析这种格式：

```text
HTTP GET /api/article/page?page=1&size=8 -> 200 (22 ms, ip=127.0.0.1)
```

解析字段：

```text
method
path
status
cost
ip
time
```

### 进阶实现

后续可以扩展：

```text
定时解析日志并落库
按天生成运行日报
慢接口阈值配置
ERROR 告警
日志文件滚动读取
支持历史 gzip 日志分析
接入 Prometheus / Grafana
接入 ELK / OpenSearch
```

### 面试可讲点

```text
操作日志、运行日志、行为埋点有什么区别
为什么不能只看业务数据库
如何从日志文件解析接口耗时
如何定义慢请求
如何统计 WARN / ERROR
如何判断系统是否正常启动和关闭
为什么运行日志分析适合做运维监控
为什么第一版不直接上 ELK
```

## 六、为什么埋点系统值得做

埋点系统算亮点，而且比单纯硬加 LLM 更稳。

原因：

```text
它贴近真实业务系统
它能体现数据意识
它能体现异步处理、去重统计、数据建模
它能自然引出 Redis、MQ、ClickHouse 等扩展方案
它能和博客前台访问行为强绑定
```

埋点系统不是简单的 `view_count + 1`，而是完整的事件采集系统。

## 七、行为埋点系统设计

### 目标

采集用户在前台的关键行为，用于后续数据分析。

### 支持事件

第一版支持：

```text
page_view：页面访问
article_view：文章浏览
article_read_duration：文章阅读时长
tag_click：标签点击
collection_click：合集点击
search：搜索行为
```

后续可以扩展：

```text
comment_submit：评论提交
like_click：点赞
share_click：分享
scroll_depth：阅读滚动深度
```

### 前端上报字段

```text
event_type
page_url
referrer
article_id
tag_id
collection_id
keyword
duration
device
browser
os
timestamp
```

### 后端记录字段

建议新增表：

```text
event_log
- id
- event_type
- user_id
- visitor_id
- article_id
- tag_id
- collection_id
- keyword
- page_url
- referrer
- ip
- user_agent
- device
- browser
- os
- duration
- create_time
```

游客没有登录时，可以使用前端生成的 `visitor_id` 标识访客。

### 接口设计

```text
POST /api/event/report
```

请求示例：

```json
{
  "eventType": "article_view",
  "articleId": 12,
  "pageUrl": "/article/12",
  "referrer": "/",
  "visitorId": "v_8f3a9c1d",
  "duration": 0
}
```

### MVP 实现

第一版不要上复杂架构，建议：

```text
前端发送事件
后端接收事件
异步线程池写 MySQL
后台按天统计 PV、UV、热门文章
```

### 进阶实现

第二版引入 Redis：

```text
Redis 计数 PV
Redis Set 去重 UV
Redis ZSet 统计热门文章
定时任务聚合落库
```

第三版可扩展为：

```text
Kafka / RabbitMQ 削峰
ClickHouse 存明细事件
后台看板做多维分析
```

### 面试可讲点

```text
PV 和 UV 有什么区别
如何用 visitor_id 识别游客
为什么不能每次访问都同步更新 MySQL
如何统计阅读时长
如何防止重复上报
如何防止刷访问量
高并发下如何用 MQ 削峰
为什么 ClickHouse 适合日志和埋点分析
```

## 八、数据分析看板

### 目标

把埋点数据转化为后台可视化分析能力。

### 看板指标

第一版：

```text
今日 PV
今日 UV
文章总浏览量
热门文章 Top 10
搜索关键词 Top 10
最近 7 天访问趋势
```

第二版：

```text
平均阅读时长
文章完读率
来源页面分布
标签点击排行
合集访问排行
设备 / 浏览器分布
```

### 后台页面

建议新增：

```text
数据概览页
文章分析页
搜索分析页
访问明细页
```

### 面试可讲点

```text
明细数据和聚合数据为什么要分开
实时统计和离线统计有什么区别
热门文章如何计算
阅读时长如何上报
如何处理用户直接关闭页面导致上报失败
```

## 九、LLM 是否算亮点

算，但不能只是简单调用 API。

如果只是：

```text
点击按钮 -> 调 LLM -> 生成摘要
```

亮点不强。

LLM 必须融入内容生产流程，体现工程化接入。

## 十、AI 内容辅助设计

### 优先做的小闭环

建议先做：

```text
AI 摘要生成
AI 标签推荐
```

暂时不要优先做 RAG。RAG 更复杂，容易拖慢主线。

### AI 摘要生成

场景：

```text
后台写完文章
点击生成摘要
LLM 根据标题和 Markdown 正文生成摘要
作者人工确认
保存到 article.summary
```

接口建议：

```text
POST /admin/ai/article/summary
```

请求：

```json
{
  "title": "Redis 缓存穿透、击穿、雪崩总结",
  "contentMd": "..."
}
```

返回：

```json
{
  "summary": "本文总结 Redis 缓存穿透、击穿、雪崩的产生原因、常见解决方案和工程实践。"
}
```

### AI 标签推荐

场景：

```text
后台写文章
点击推荐标签
LLM 根据文章内容推荐已有标签
作者选择后保存
```

接口建议：

```text
POST /admin/ai/article/tags
```

返回：

```json
{
  "keywords": ["Redis", "缓存", "高并发"],
  "recommendedTagIds": [1, 3]
}
```

### AI 调用日志

为了体现工程化，不要只调接口。建议新增：

```text
ai_call_log
- id
- scene
- model
- prompt
- result
- prompt_tokens
- completion_tokens
- success
- error_message
- cost_time
- create_time
```

### 关键工程点

```text
Prompt 模板管理
超长文章截断或分段摘要
调用失败重试
超时处理
调用日志记录
AI 结果人工确认后再入库
敏感信息过滤
```

### 面试可讲点

```text
为什么 AI 结果不能直接发布
如何设计 Prompt
如何处理超长文章
如何记录 AI 调用成本
如何处理调用失败
如何把 AI 能力接入现有文章发布流程
```

## 十一、RAG 文章问答是否要做

RAG 是强亮点，但不建议马上做。

原因：

```text
需要文档切分
需要向量化
需要向量数据库
需要检索召回
需要 Prompt 拼接
需要引用来源
工程量明显大于 AI 摘要和标签推荐
```

建议放在最后：

```text
先完成 AI 摘要和标签推荐
再完成搜索系统
最后再做 RAG 文章问答
```

如果后续做 RAG，定位可以是：

```text
基于个人博客文章库的智能问答助手
```

可支持：

```text
问某篇文章内容
问博客里所有 Redis 相关文章
回答时附带引用文章链接
```

## 十二、最终推荐开发顺序

按秋招展示价值和实现稳定性排序：

```text
1. 前台展示 MVP 端到端验收
2. MinIO 文件上传
3. 操作日志系统
4. 运行日志分析系统 MVP
5. 行为埋点系统 MVP
6. 数据分析看板 MVP
7. AI 摘要生成
8. AI 标签推荐
9. 搜索系统
10. Redis 缓存优化
11. RAG 文章问答
```

不要跳过前 3 步直接做 AI。否则项目会变成“基础不稳但强行加热点”。

## 十三、简历亮点写法

### MinIO 文件上传

```text
基于 MinIO 实现文章封面和 Markdown 图片上传，完成文件类型校验、大小限制、对象命名和 URL 持久化，避免大文件直接进入数据库。
```

### 行为埋点

```text
设计并实现前台行为埋点系统，支持页面访问、文章浏览、搜索、标签点击、阅读时长等事件采集，后端采用异步写入降低接口响应耗时，并结合 Redis 完成 PV/UV 去重统计和热门文章排行。
```

### 数据看板

```text
基于埋点数据构建后台数据分析看板，支持 PV/UV 趋势、热门文章、搜索关键词、标签点击排行等指标展示，为内容优化提供数据反馈。
```

### AI 内容辅助

```text
接入大模型能力实现文章摘要生成与标签推荐，通过场景化 Prompt、调用日志、失败处理和人工确认机制，将 AI 结果纳入后台内容发布流程。
```

### 操作日志

```text
基于自定义注解和 Spring AOP 实现后台操作日志，自动记录管理员、请求路径、参数、耗时、异常信息，并通过异步写入降低对业务接口响应时间的影响。
```

### 运行日志分析

```text
基于后端运行日志文件实现轻量级日志分析模块，支持请求日志解析、WARN/ERROR 统计、慢接口分析、接口访问排行和系统启动关闭状态识别，用于辅助后台运维监控和问题排查。
```

## 十四、面试表达方式

当面试官说“博客项目很普通，你这个有什么亮点”时，可以这样回答：

```text
这个项目最开始确实是博客形态，但我后续把它往内容平台方向做了扩展。除了基础的文章发布和前台展示，我重点做了三个工程模块：

第一是对象存储，使用 MinIO 支持文章封面和 Markdown 图片上传；
第二是日志和数据分析能力，包括后台操作日志、运行日志分析和前台行为埋点，用来支持后台审计、运维排查和内容数据反馈；
第三是 AI 内容辅助，支持根据文章内容生成摘要和推荐标签，并且有调用日志和人工确认流程。

所以它不是单纯 CRUD 博客，而是一个包含内容生产、内容消费和数据反馈闭环的内容管理平台。
```

## 十五、最终目标

最终项目要形成这个闭环：

```text
后台写文章
-> 上传图片
-> AI 生成摘要和推荐标签
-> 作者确认发布
-> 前台读者阅读
-> 埋点采集行为
-> 后台数据看板分析内容表现
-> 作者根据数据优化后续内容
```

这个闭环才是项目去同质化的关键。

最终对外介绍：

```text
kyonelife 是一个数据驱动的智能内容平台，支持内容管理、前台展示、对象存储、行为埋点、数据分析和 AI 内容辅助能力。
```
