# kyonelife 博客项目部署计划

## 一、部署目标

将本地已经完成测试的博客项目部署到服务器，并通过自己的域名访问。

目标访问方式：

```text
前台博客：https://你的域名/
后台管理：https://你的域名/admin/
后端接口：https://你的域名/api/...
```

## 二、部署前提

部署前建议先完成：

- 前台展示 MVP 手动测试通过
- 后台能登录
- 后台能新建、编辑、发布、下架文章
- 前台首页能看到已发布文章
- 前台详情、标签、合集、归档能正常访问
- 草稿和下架文章不会出现在前台
- `blog-web` 可以正常执行 `npm run build`
- `blog-admin` 可以正常执行 `npm run build`
- `blog` 可以正常打包为 jar

## 三、服务器准备

建议准备一台云服务器，例如阿里云、腾讯云、华为云等。

最低配置建议：

```text
CPU：1 核或 2 核
内存：2GB 起步
系统：Ubuntu 22.04 / CentOS / Debian 均可
带宽：1Mbps 起步
```

服务器需要安装：

- JDK 21
- MySQL 8
- Nginx
- Git
- Node.js，可选。如果前端在本地构建后上传，则服务器不一定需要 Node.js

## 四、推荐目录规划

服务器上建议使用以下目录：

```text
/opt/kyonelife
  /backend
    blog.jar
    application-prod.yml
  /frontend
    /blog-web
    /blog-admin
  /logs
```

说明：

- `/backend`：放后端 jar 和生产配置
- `/frontend/blog-web`：放前台构建后的静态文件
- `/frontend/blog-admin`：放后台构建后的静态文件
- `/logs`：放后端运行日志

## 五、数据库部署计划

### 1. 创建数据库

数据库名建议保持和本地一致：

```text
kyonelife
```

### 2. 初始化表结构

使用项目中的初始化脚本：

```text
blog/sql/init.sql
```

执行前确认脚本里的数据库名和后端配置一致。

### 3. 创建生产数据库账号

不建议生产环境直接使用 root 账号。

建议创建单独账号，例如：

```text
用户名：kyonelife_user
权限：只允许访问 kyonelife 数据库
```

## 六、后端部署计划

### 1. 本地打包后端

进入后端目录：

```bash
cd blog
./mvnw clean package -DskipTests
```

打包后得到：

```text
blog/target/blog-0.0.1-SNAPSHOT.jar
```

### 2. 上传到服务器

上传到：

```text
/opt/kyonelife/backend/blog.jar
```

### 3. 配置生产环境

建议准备生产配置文件：

```text
/opt/kyonelife/backend/application-prod.yml
```

至少需要配置：

- MySQL 地址
- MySQL 用户名
- MySQL 密码
- Redis 地址，如果生产环境实际使用 Redis
- 日志路径
- 端口，例如 `8080`

### 4. 启动后端

可以先用普通命令启动：

```bash
java -jar blog.jar --spring.profiles.active=prod
```

正式部署建议使用 `systemd` 管理，让服务支持开机自启和异常重启。

后端建议只监听服务器本机端口：

```text
127.0.0.1:8080
```

外部访问通过 Nginx 反向代理，不直接暴露 8080 端口。

## 七、前台部署计划

### 1. 构建前台博客

进入前台目录：

```bash
cd blog-web
npm run build
```

构建产物一般在：

```text
blog-web/dist
```

上传 `dist` 里的内容到服务器：

```text
/opt/kyonelife/frontend/blog-web
```

### 2. 构建后台管理端

进入后台目录：

```bash
cd blog-admin
npm run build
```

构建产物一般在：

```text
blog-admin/dist
```

上传 `dist` 里的内容到服务器：

```text
/opt/kyonelife/frontend/blog-admin
```

## 八、Nginx 部署计划

Nginx 负责三件事：

- 托管前台静态页面
- 托管后台静态页面
- 把 `/api` 和 `/admin` 接口请求代理到后端

推荐路由规划：

```text
https://你的域名/         -> blog-web 前台
https://你的域名/admin/   -> blog-admin 后台页面
https://你的域名/api/...  -> 后端前台接口
https://你的域名/admin-api/... 或 /admin/... -> 后端后台接口
```

注意：

如果后台页面也放在 `/admin/`，而后端后台接口也使用 `/admin/...`，Nginx 配置时需要避免路径冲突。

更清晰的方式是：

```text
后台页面：https://你的域名/admin/
后端后台接口：https://你的域名/admin-api/...
```

然后在 Nginx 中把 `/admin-api/` 转发到后端实际的 `/admin/` 接口。

## 九、域名解析计划

在域名服务商控制台添加 DNS 解析：

```text
类型：A 记录
主机记录：@
记录值：服务器公网 IP
```

如果还想支持 `www`：

```text
类型：A 记录
主机记录：www
记录值：服务器公网 IP
```

解析生效后，可以通过域名访问服务器。

## 十、HTTPS 计划

正式上线建议必须配置 HTTPS。

可以使用：

- 云厂商免费 SSL 证书
- Let's Encrypt 免费证书

HTTPS 配置完成后，最终访问地址应为：

```text
https://你的域名/
```

同时建议把 HTTP 自动跳转到 HTTPS。

## 十一、上线验收清单

部署完成后，需要测试：

- 域名可以打开前台首页
- 前台首页能看到已发布文章
- 点击文章能进入详情页
- 标签页能打开
- 合集页能打开
- 归档页能打开
- 后台登录页能打开
- 后台能登录
- 后台能发布文章
- 发布后前台能看到文章
- 下架后前台看不到文章
- 刷新任意前端路由不会 404
- 手机访问没有明显布局错乱
- HTTPS 证书有效

## 十二、后续修改功能是否需要重新部署

结论：大多数代码功能修改后，都需要重新部署受影响的部分。

### 1. 只修改前台博客页面

例如修改：

- `blog-web` 页面
- `blog-web` 样式
- `blog-web` 路由
- `blog-web` 请求逻辑

需要重新执行：

```bash
cd blog-web
npm run build
```

然后把新的 `dist` 上传覆盖服务器上的：

```text
/opt/kyonelife/frontend/blog-web
```

后端不需要重启。

### 2. 只修改后台管理页面

例如修改：

- `blog-admin` 页面
- `blog-admin` 样式
- `blog-admin` 请求逻辑

需要重新执行：

```bash
cd blog-admin
npm run build
```

然后把新的 `dist` 上传覆盖服务器上的：

```text
/opt/kyonelife/frontend/blog-admin
```

后端不需要重启。

### 3. 修改后端接口或业务逻辑

例如修改：

- Controller
- Service
- Mapper
- Entity
- DTO / VO
- 权限逻辑
- 文章、标签、合集、评论等业务逻辑

需要重新打包后端：

```bash
cd blog
./mvnw clean package -DskipTests
```

然后上传新的 jar，并重启后端服务。

前端不一定需要重新部署，除非接口字段、接口路径或前端调用方式也变了。

### 4. 修改数据库表结构

例如新增字段、新增表、修改字段类型。

需要：

- 执行数据库变更 SQL
- 重新部署后端
- 如果前端页面也使用了新字段，还要重新部署对应前端

这类修改上线前要特别小心，建议先备份数据库。

### 5. 只修改文章内容

如果只是通过后台：

- 新增文章
- 编辑文章
- 发布文章
- 下架文章
- 修改标签
- 修改合集

不需要重新部署。

这些是数据库内容变化，前台会通过接口读取最新数据。

### 6. 只修改 Nginx 配置

例如修改：

- 域名
- HTTPS 证书
- 反向代理路径
- 静态文件目录

不需要重新打包项目。

只需要重新加载 Nginx：

```bash
nginx -s reload
```

## 十三、推荐上线节奏

建议按这个顺序做：

```text
1. 本地手动测试通过
2. 修复发现的问题
3. 后端打包
4. 前台 blog-web 打包
5. 后台 blog-admin 打包
6. 服务器安装 Java / MySQL / Nginx
7. 初始化数据库
8. 上传后端 jar 和前端 dist
9. 配置后端生产环境
10. 配置 Nginx
11. 配置域名解析
12. 配置 HTTPS
13. 用域名完整验收
```

## 十四、下一步建议

在真正部署前，建议先补一份更具体的服务器部署记录，记录：

- 服务器 IP
- 域名
- MySQL 数据库名
- 后端端口
- 前台部署目录
- 后台部署目录
- Nginx 配置文件位置
- 后端 systemd 服务名

这样后续功能修改和重新部署时，不容易忘记服务器上的实际路径和命令。
