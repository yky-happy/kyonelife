## 二、核心决策：推进顺序

### 方案一：纵向切片（推荐）

> 每次完成一个**端到端**的完整功能，能跑起来再进入下一个。

```
顺序：
1. 打通后台登录流程（后端已就绪 → 搭前端框架 + 登录页 → 联调拿到 Token）
2. 标签 CRUD（后端 2 个接口 → 前端标签管理页 → 联调）
3. 合集 CRUD（同上）
4. 文章 CRUD（后端含多标签关联 → 前端含 Markdown 编辑器 → 联调）
5. 前台 blog-web 开发（文章列表/详情，依赖上一步 api/ 接口）
```

**优点**
- 每完成一步都有可运行的结果，能持续验证整体链路
- 前端框架搭好后可以快速复用，后续每个功能只需套模板
- 阻塞点早暴露（例如 Sa-Token 跨域、Token 存储方式），不会到最后才发现

**缺点**
- 前几天需要同时写后端和前端，上下文切换多
- 前端框架搭建（Layout/Router/Pinia）有一次性投入成本

**适合**：想边做边验证效果、看到进展有动力的节奏。

---

### 方案二：后端优先（横向铺开）

> 先把后台所有管理接口全部完成，再统一开发前端。

```
顺序：
1. 标签 / 合集 / 文章 / 评论 / Banner / WebConfig CRUD（全部后端）
2. 动态菜单接口（登录后返回角色菜单树）
3. 统一开始前端，所有接口已就绪，前端直接对接
```

**优点**
- 思维不用在前后端之间切换，后端写起来流畅
- 后端全量完成后，用 Knife4j 文档可以系统性地验证所有接口
- 前端开发期间后端接口稳定，不会出现接口变更导致前端返工

**缺点**
- 前几周只有接口文档，没有可视化界面，进度不直观
- 真正联调时问题会集中爆发（跨域、字段名、分页参数格式……）
- 动态菜单接口的数据结构到底怎么设计，不联调前很难确认是否对前端友好

**适合**：后端思维更强、不需要看到 UI 就能保持节奏的情况。

---

### 方案三：前端先行（Mock 数据）

> 先把 blog-admin 前端框架和所有页面骨架搭完，用 Mock 数据调试 UI，最后再对接后端。

```
顺序：
1. 搭 blog-admin 框架（Router 动态路由、Pinia、axios 拦截器、Layout 组件）
2. 搭各管理页面骨架 + Mock 数据（vite-plugin-mock）
3. 后端接口逐步完成，替换 Mock
```

**优点**
- 前端 UI 和交互可以独立快速迭代，不依赖后端进度
- Mock 数据让边界情况（空列表、分页末页、长文本）容易测试

**缺点**
- 需要维护一套 Mock 数据，接口结构变化时 Mock 也要跟着改，是额外工作量
- 容易出现 Mock 通了但真实接口结构不一致的隐患
- 目前只有一个人开发，前后端不需要解耦进度

**适合**：前后端分工明确、需要并行推进的团队场景；个人项目意义不大。

## 三、技术选型决策

无论选哪种推进顺序，以下 4 个选型都需要在开始写代码前确认。

---

### 3.1 文章 Markdown 编辑器

这是文章 CRUD 中最有影响的技术决策，一旦集成后换起来成本高。

#### 选项 A：md-editor-v3

```
特点：Vue3 专属，API 设计贴合 Vue3 生态，文档完善
支持：Markdown + 实时预览（左右分屏），代码高亮，图片粘贴上传，目录
bundle size：~500KB（含 highlight.js）
```

**优点**：Vue3 原生，组件式调用简单，图片上传钩子 `onUploadImg` 直接对接后端接口<br>
**缺点**：


---

#### 选项 B：vditor

```
特点：知乎/思否同款，支持三种模式：Markdown / WYSIWYG / IR（即时渲染）
支持：数学公式、流程图、思维导图、图片上传、代码高亮
bundle size：~2MB（功能多，体积大）
```

**优点**：功能最全，三种编辑模式可切换，写技术博客非常顺手<br>
**缺点**：Vue3 集成需要手动 `import`，体积大，配置项多学习成本略高

---

#### 选项 C：bytemd（掘金同款）

```
特点：掘金编辑器同款，插件化架构，轻量
Vue3 包：@bytemd/vue-next
bundle size：核心 ~100KB，按需加载插件
```

**优点**：最轻量，插件化可按需选择高亮/数学公式/目录，掘金同款品质有保证<br>
**缺点**：社区相对小，Vue3 适配文档少，遇到问题排查成本高

---

**选型建议**：个人博客写技术文章选 **vditor**，功能体验最好；想快速上手、bundle 体积敏感选 **md-editor-v3**。

---

### 3.2 图片存储方案

图片上传影响文章编辑器的集成方式和后端 Controller 设计，也需要提前确认。

#### 选项 A：本地存储（MVP 推荐）

```
后端：MultipartFile 接收 → 存到服务器本地目录 → 返回访问 URL
配置：Spring MVC 静态资源映射 /upload/** → /var/blog/upload/
```

**优点**：零费用，零外部依赖，实现最简单（10 行代码）<br>
**缺点**：服务器重装/迁移会丢图片，不适合长期生产使用

---

#### 选项 B：MinIO（自建 OSS）

```
部署：Docker 启动 MinIO 容器（本机或服务器）
后端：minio-java-sdk，上传文件返回 presigned URL
配置：bucket 策略设为公开读
```

**优点**：完全自主可控，兼容 S3 API，后续迁移到阿里云 OSS 只需改配置<br>
**缺点**：本地还需要多跑一个 MinIO 容器，开发环境多一个依赖项

---

#### 选项 C：阿里云 OSS / 腾讯云 COS

```
后端：OSS/COS SDK，上传返回公网 URL
费用：存储 ~0.12元/GB/月，流量按量计费，博客量级可忽略不计
```

**优点**：生产级稳定，CDN 加速，不用运维存储本身<br>
**缺点**：需要花钱买账号/存储桶，本地开发也依赖外网

---

**选型建议**：MVP 阶段用**本地存储**先跑通流程；上线或想把简历写得好看时切换到 **MinIO**（加一项"自建对象存储"的技术点）。两者切换成本极低，只需替换 Service 实现类。

---

### 3.3 阅读量统计架构

文章详情接口调用时触发，设计不同性能差距很大。

#### 选项 A：请求时直接 UPDATE MySQL

```sql
UPDATE article SET view_count = view_count + 1 WHERE id = ?
```

**优点**：实现最简单，数据实时准确<br>
**缺点**：高并发时 MySQL 行锁争用，对热门文章影响写性能（博客流量小时可接受）

---

#### 选项 B：Redis INCR + 定时刷库（推荐）

```
读文章时：INCR article:view:${id}
定时任务（每5分钟）：把 Redis 中所有计数批量 UPDATE 回 MySQL，然后清零
```

**优点**：Redis 原子操作无锁，性能极好；定时批量更新减少 MySQL 写压力；是标准的简历技术亮点<br>
**缺点**：服务重启时 Redis 中未刷库的计数会丢失（可接受，用 AOF 持久化可缓解）

---

**选型建议**：已经引入了 Redis 依赖，直接用**方案 B**，几乎不增加工作量，还能在简历上写"Redis 缓存 + 定时刷库"。

---

### 3.4 前端动态路由方案（影响后端菜单接口设计）

后端登录后返回的菜单树，直接决定前端动态路由怎么注册，需要提前对齐格式。

#### 选项 A：后端返回完整树形菜单，前端映射为路由

```json
[
  {
    "id": 1, "title": "内容管理", "type": "CATALOG",
    "children": [
      { "id": 10, "title": "文章管理", "type": "MENU", "path": "/content/article", "component": "content/article/index" }
    ]
  }
]
```

前端用 `router.addRoute()` 动态注册，`component` 字段用于 `import()` 懒加载。

**优点**：权限完全由后端控制，前端只是渲染；菜单变更不需要部署前端<br>
**缺点**：前端需要建立 `component` 字符串到实际组件的映射表，多一步转换逻辑

---

#### 选项 B：前端静态路由，后端只返回权限列表

```json
["article:list", "article:add", "tag:list", "tag:add"]
```

前端根据权限列表决定按钮是否显示，路由直接写死在 router/index.ts。

**优点**：前端路由配置直观，不需要动态注册路由<br>
**缺点**：菜单结构写死在前端，后端新增菜单必须同步修改前端代码并重新部署

---

**选型建议**：用**方案 A**，后端已经设计了完整的菜单树结构，走动态路由是更正规也更有技术含量的做法。

---

## 四、下一步任务清单（按纵向切片推进）

基于方案一（纵向切片），最推荐的下一步顺序：

### 阶段 1：打通登录联调（预计 1 天）

**后端（已就绪，无需开发）**

**前端 blog-admin 框架搭建**：
- `main.ts`：挂载 Element Plus、Router、Pinia
- `router/index.ts`：登录页（静态路由）+ 布局页（待动态路由填充）
- `stores/auth.ts`：Pinia 存储 Token、用户信息

- `views/login/index.vue`：登录表单，调用 POST /admin/auth/login，成功后跳转 dashboard

---

### 阶段 2：动态菜单接口 + 前端动态路由（预计 1 天）

**后端**：
- `admin/controller/MenuController.java`：GET /admin/menu/tree，根据当前登录管理员角色返回菜单树（仅 CATALOG + MENU 类型，不含 BUTTON）
- `admin/controller/MenuController.java`：GET /admin/menu/perms，返回当前用户所有按钮权限标识列表（供前端控制按钮显示）

**前端**：
- 登录成功后请求菜单树，`router.addRoute()` 动态注册路由
- Layout 组件：左侧 Sidebar（渲染菜单树）+ 顶部 Header（用户信息 + 退出登录）

---

### 阶段 3：标签 + 合集 CRUD（预计 1 天）

结构简单，没有关联关系，适合练熟后端 → 前端 → 联调的完整流程。

**后端**：
- `admin/controller/TagController.java`：列表（带分页）/ 新增 / 编辑 / 删除
- `admin/controller/CollectionController.java`：同上

**前端**：
- `views/content/tag/index.vue`：El-Table + El-Dialog 表单
- `views/content/collection/index.vue`：同上

---

### 阶段 4：文章 CRUD（预计 3～4 天）

最复杂的模块，建议拆成两个子阶段：

**4a：文章列表 + 基础字段（先不含编辑器和标签关联）**
- GET /admin/article/page：分页列表（含状态筛选、标题搜索）
- DELETE /admin/article/{id}
- PUT /admin/article/{id}/status：修改状态（发布/下架/草稿）
- 前端：文章列表页

**4b：文章新增/编辑（含 Markdown 编辑器 + 多标签关联）**
- POST /admin/article：新增，接收标签 ID 列表，事务内批量插 article_tag
- PUT /admin/article/{id}：编辑，先删旧标签关联再插新的
- 前端：文章编辑页（接入 Markdown 编辑器 + 标签多选）

---

**后端 api/ 模块**：
- GET /web/article/page：前台文章列表（只返回已发布文章）
- GET /web/article/{id}：文章详情（同时触发阅读量 Redis INCR）
- GET /web/collection/list：合集列表
- GET /web/tag/list：标签列表

## 五、SSR vs SPA：blog-web 的一个额外决策

这不是现在必须决定的，但早点想清楚能避免后期重构。

### 方案 A：继续用 Vite + Vue3 SPA

**优点**：和 blog-admin 同一套技术，学习成本低，开发快<br>
**缺点**：搜索引擎爬虫无法抓取内容，SEO 几乎为零；首屏要等 JS 加载完才渲染

### 方案 B：改用 Nuxt 3（SSR/SSG）

**优点**：服务端渲染，搜索引擎可直接抓取 HTML，SEO 好；首屏速度快<br>
**缺点**：需要重新搭一个 Nuxt 项目，部署需要 Node 服务（不能只用静态 CDN）

### 方案 C：Vite + Vue3 + 预渲染（折中）

**优点**：构建时生成静态 HTML，SEO 和首屏兼顾；部署同 SPA<br>
**缺点**：动态内容（评论、阅读量）无法预渲染，只能 CSR

**建议**：博客的核心价值之一是被搜索引擎收录。如果在乎 SEO，blog-web 应该用 **Nuxt 3**，趁现在还没写业务代码切换成本为零。如果主要是练手、不在乎 SEO，继续 Vite SPA 更快。

## 六、决策清单（开始写代码前需要确认的）

| 决策项 | 选项 | 建议 |
|--------|------|------|
| 推进顺序 | 纵向切片 / 后端优先 / 前端先行 | **纵向切片** |
| Markdown 编辑器 | md-editor-v3 / vditor / bytemd | **vditor**（功能最好）或 **md-editor-v3**（最简单） |

| 动态路由 | 后端返回树 / 前端静态路由 | **后端返回树**（方案 A）|
| blog-web 渲染 | SPA / Nuxt SSR / 预渲染 | 看是否在乎 SEO，建议 **Nuxt 3** |
