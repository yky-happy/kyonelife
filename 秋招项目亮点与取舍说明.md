# kyonelife 秋招项目亮点与取舍说明

## 一、项目定位

`kyonelife` 不再按普通博客系统去包装，而是定位为：

```text
数据驱动的智能内容管理与展示平台
```

核心表达：

```text
它不是只做文章 CRUD，而是围绕内容生产、内容展示、用户行为反馈、运行状态监控和 AI 辅助创作形成闭环。
```

## 二、核心亮点是否够用

如果最终完成以下四个模块，作为秋招项目亮点是够用的：

```text
1. 行为埋点系统
2. 数据分析看板
3. 运行日志分析系统
4. AI 内容辅助
```

这四个模块比普通评论系统更能体现工程能力。

关键不是功能数量，而是每个模块要做出工程细节。

## 三、四个核心亮点怎么做才有技术含量

### 行为埋点系统

不要只做 `viewCount + 1`。

应该体现：

```text
前端事件上报
visitorId 游客标识
文章浏览事件
标签点击事件
合集点击事件
搜索事件
阅读时长统计
异步写入
PV / UV 去重统计
```

进阶增强：

```text
批量上报
离线缓存
失败重试
本地队列限流
重复事件去重
```

批量上报设计：

```text
前端先把事件写入本地队列
达到一定数量或固定时间间隔后批量上报
后端批量接收事件
后端异步写入 MySQL
```

离线缓存设计：

```text
网络失败时将事件缓存到 localStorage / IndexedDB
下次页面打开时尝试补发
浏览器恢复网络时补发
限制本地缓存最大数量，避免无限增长
```

接口可以设计为：

```text
POST /api/event/batch
```

请求示例：

```json
{
  "visitorId": "v_8f3a9c1d",
  "events": [
    {
      "eventType": "article_view",
      "articleId": 4,
      "pageUrl": "/article/4",
      "timestamp": "2026-06-02 16:20:00"
    }
  ]
}
```

面试表达：

```text
我没有直接在文章表上简单累加阅读量，而是设计了事件上报模型，将用户访问、点击、搜索、阅读时长等行为抽象成事件，后端异步写入，再基于明细数据做统计分析。
```

增强版面试表达：

```text
为了降低频繁请求对前后端的压力，我在前端设计了事件队列，支持批量上报和离线缓存。网络失败时事件先缓存在本地，网络恢复后再补发，后端批量接收并异步写入，从而兼顾用户体验和数据完整性。
```

### 数据分析看板

不要只展示几个总数。

应该体现：

```text
今日 PV
今日 UV
热门文章 Top 10
搜索关键词 Top 10
标签点击排行
合集访问排行
最近 7 天访问趋势
文章阅读时长
```

进阶增强：

```text
定时聚合任务
明细表和聚合表分离
看板优先查询聚合表
每天凌晨补算昨日数据
聚合任务幂等更新
```

建议数据表：

```text
event_log：埋点明细表
event_daily_stat：每日访问聚合表
article_daily_stat：文章每日统计表
search_keyword_stat：搜索关键词统计表
```

定时任务设计：

```text
每 5 分钟聚合一次今日数据
每天凌晨补算昨天数据
聚合结果写入统计表
看板查询统计表，不直接扫描明细表
```

关键工程点：

```text
Spring Scheduler 定时任务
按天聚合
幂等更新
明细数据和聚合数据分离
实时统计和离线统计取舍
```

面试表达：

```text
数据看板不是直接查业务表，而是基于埋点明细和聚合统计，为内容运营提供反馈，比如哪些文章更受欢迎、哪些关键词搜索更多、哪些标签点击更高。
```

增强版面试表达：

```text
看板没有每次直接扫描埋点明细表，而是通过定时任务把明细数据聚合到统计表中。这样可以降低大数据量下的查询压力，同时通过幂等更新保证定时任务重复执行时结果稳定。
```

### 运行日志分析系统

不要只做“查看日志文件”。

应该体现：

```text
解析后端 log 文件
统计今日请求总数
统计 WARN / ERROR 数量
识别慢请求
统计接口访问排行
统计接口平均耗时
识别后端启动和关闭时间
辅助判断系统运行状态
```

面试表达：

```text
我基于后端运行日志实现了轻量级日志分析模块，可以从请求日志中解析接口路径、状态码、耗时和 IP，并统计慢接口、异常数量和系统启动关闭状态，用于辅助运维排查。
```

### AI 内容辅助

不要只做“按钮调用大模型”。

应该体现：

```text
AI 摘要生成
AI 标签推荐
Prompt 模板
调用日志
调用结果缓存
失败处理
超时处理
降级逻辑
人工确认
AI 结果保存到文章发布流程
```

调用结果缓存设计：

```text
根据 scene + title + contentHash 生成 cacheKey
优先查询 Redis 缓存
命中缓存时直接返回结果
未命中时调用 AI
调用成功后写入缓存和 ai_call_log
```

示例：

```text
summary:article:{contentHash}
tags:article:{contentHash}
```

降级逻辑设计：

```text
AI 调用超时：提示稍后重试
AI 调用失败：返回最近一次成功结果
摘要生成失败：使用本地规则截取正文前几百字作为临时摘要
标签推荐失败：使用关键词匹配已有标签
AI 服务不可用：保留人工填写入口
```

面试表达：

```text
AI 结果不会直接发布，而是进入后台内容生产流程，由作者人工确认后再保存，避免模型幻觉和不可控输出。
```

增强版面试表达：

```text
AI 调用成本和稳定性都需要考虑，所以我设计了调用结果缓存和降级逻辑。同一篇文章重复生成摘要或标签时优先命中缓存，避免重复消耗 token；当模型超时或不可用时，系统会返回最近一次成功结果或使用本地规则降级，保证后台编辑流程不中断。
```

## 四、可以不做评论系统

评论系统不是当前项目的核心亮点，可以不做。

原因：

```text
垃圾评论处理复杂
需要敏感词过滤
需要评论审核
需要用户身份体系
需要楼中楼或回复关系
需要后台删除和举报能力
和当前数据驱动内容平台定位不强相关
```

当前项目可以只保留轻量互动：

```text
文章访问
文章点赞
阅读时长
标签点击
合集点击
搜索行为
```

面试表达：

```text
第一阶段没有做评论系统，是因为项目定位更偏个人内容管理和数据反馈，评论会引入审核、垃圾内容和用户关系链等额外复杂度。当前用户互动通过点赞、阅读时长、搜索和点击行为完成，并进入数据分析看板。
```

## 五、点赞功能是否可以做

可以做，但不要把点赞作为核心亮点。

点赞适合作为轻量互动和埋点事件的一部分。

建议实现：

```text
文章点赞
取消点赞
游客 visitorId 防重复点赞
登录用户 userId 防重复点赞
点赞数展示
点赞事件进入埋点系统
```

面试表达：

```text
点赞不是单独做一个简单计数，而是和 visitorId、用户行为事件、数据看板结合起来，作为内容反馈指标之一。
```

## 六、是否可以加入旅游规划 Agent

可以加入，但不要让它抢走主项目定位。

不建议包装成：

```text
博客系统 + 旅游规划 Agent
```

这样会显得业务发散。

更合理的定位是：

```text
旅行灵感助手
```

它属于内容平台里的 AI 创作辅助模块。

推荐闭环：

```text
输入目的地、天数、预算、偏好
-> AI 生成旅行规划
-> 转成 Markdown 攻略
-> 保存为文章草稿
-> 后台编辑确认
-> 发布到前台文章
```

这样它不是一个孤立 Demo，而是服务于内容生产。

## 七、旅游规划 Agent 的优先级

旅游规划 Agent 可以做，但优先级低于核心模块。

推荐顺序：

```text
1. 前台展示 MVP
2. MinIO 文件上传
3. 操作日志系统
4. 运行日志分析系统
5. 数据看板定时聚合任务
6. 行为埋点系统
7. 埋点批量上报和离线缓存
8. 数据分析看板
9. AI 摘要生成
10. AI 标签推荐
11. AI 调用结果缓存和降级逻辑
12. 旅行灵感助手
```

如果时间不够，旅行灵感助手可以只作为扩展规划，不作为核心交付。

## 八、最终项目亮点表达

简历可以这样写：

```text
基于 Spring Boot + Vue 实现数据驱动的智能内容管理与展示平台，除文章、标签、合集等内容管理能力外，重点建设了行为埋点、数据分析看板、运行日志分析和 AI 内容辅助模块，形成内容生产、内容消费、数据反馈和系统监控闭环。
```

项目亮点可以拆成：

```text
行为埋点：支持页面访问、文章浏览、搜索、标签点击、阅读时长等事件采集，并支持批量上报、离线缓存和失败补发
数据看板：支持 PV/UV、热门文章、搜索关键词、标签点击排行和访问趋势分析，并通过定时聚合任务降低明细表查询压力
运行日志分析：支持请求日志解析、WARN/ERROR 统计、慢接口分析和系统启动关闭状态识别
AI 内容辅助：支持文章摘要生成、标签推荐、调用日志、结果缓存、降级逻辑和人工确认流程
```

## 九、面试回答模板

当面试官问：

```text
博客项目比较普通，你这个项目有什么区别？
```

可以回答：

```text
这个项目最开始确实是博客形态，但我没有停留在文章 CRUD，而是把它往内容平台方向扩展。

第一，内容生产侧支持后台文章管理、图片上传、AI 摘要生成和标签推荐；
第二，内容消费侧支持前台文章展示、标签、合集、归档和搜索；
第三，数据反馈侧做了行为埋点和数据看板，可以统计 PV/UV、阅读时长、热门文章和搜索关键词；
第四，工程治理侧做了操作日志和运行日志分析，可以辅助后台审计、慢接口定位和异常排查。

在实现上，我也考虑了可靠性和性能，比如埋点支持批量上报和离线缓存，数据看板通过定时任务聚合明细数据，AI 调用则加入了结果缓存和降级逻辑，避免重复消耗 token，并保证模型不可用时后台编辑流程不中断。

所以它不是单纯个人博客，而是一个包含内容生产、内容消费、数据反馈和运行监控闭环的智能内容平台。
```

## 十、取舍原则

后续开发不要盲目堆功能。

优先做能支撑项目定位的功能：

```text
能体现工程能力
能形成业务闭环
能在面试中讲清楚设计原因
能和内容平台定位强相关
```

低优先级功能：

```text
评论系统
复杂社交关系
大而全的用户中心
和内容平台无关的孤立工具页
```

最终目标：

```text
少做普通功能，多做能解释业务价值和工程设计的功能。
```
