# kyonelife 秋招展示型工程项目改进路线

## 一、项目定位

`kyonelife` 不应该只被当成一个普通博客练手项目，而应该被改造成一个能在秋招面试中展示工程能力的后端项目。

面试官真正关心的不是“你做了一个博客”，而是：

```text
你是否能设计数据模型
你是否能拆分前后台接口
你是否能处理鉴权、跨域、异常、日志、上传、搜索、缓存
你是否能解释技术选型
你是否能从 MVP 演进到更完整的工程系统
```

所以这个项目后续的目标不是堆功能，而是把每个功能做成可以讲清楚的工程模块。

## 二、当前项目状态

当前项目已经具备比较好的基础：

```text
后端：Spring Boot + MyBatis Plus + Sa-Token + Knife4j + MySQL
后台：Vue3 + Element Plus + Pinia + axios
前台：Vue3 + Vite + vue-router + axios
```

已经完成或基本完成的能力：

```text
统一响应 Result
全局异常处理
CORS 配置
Sa-Token 登录鉴权
后台登录链路
标签 CRUD
合集 CRUD
文章 CRUD
文章发布 / 下架 / 草稿状态
前台文章列表
前台文章详情
前台标签页
前台合集页
前台归档页
公开 API 与后台 API 分离
```

当前阶段判断：

```text
阶段 4：后台文章管理 MVP 已完成
阶段 5：前台展示 MVP 已基本完成
下一步重点：端到端验收 + 工程化能力增强
```

## 三、秋招展示目标

最终这个项目在简历上应该能写成：

```text
基于 Spring Boot + Vue3 开发的个人内容管理与展示平台，支持后台文章管理、标签/合集组织、前台文章展示、文件上传、操作日志、评论、搜索、缓存优化等能力。项目中完成了前后台接口隔离、统一异常处理、鉴权拦截、跨域处理、对象存储接入、AOP 操作日志、前台公开接口安全过滤等工程设计。
```

更具体一点，面试中应该能围绕下面几个模块展开讲：

```text
登录鉴权系统
文章发布系统
文件上传系统
操作日志系统
评论系统
搜索系统
缓存与性能优化
部署与可观测性
```

后续建议按下面顺序推进：

```text
阶段 5 收尾：前台展示 MVP 端到端验收
阶段 6：MinIO 文件上传系统
阶段 7：后台操作日志系统
```

## 五、阶段 5 收尾：前台展示 MVP 端到端验收

### 目标

确认当前已经写好的前台展示功能真的可用。

必须跑通：

```text
后台登录
新建标签
新建合集
写文章
保存草稿
发布文章
前台首页看到文章
点击进入文章详情
按标签查看文章
按合集查看文章
归档页看到文章
```

### 验收标准

```text
后台可以新建、编辑、发布、下架文章
前台只展示 status = 1 的已发布文章
草稿和下架文章不会出现在前台
文章详情能正确渲染 Markdown
标签、合集、归档页面能正常跳转
手机端页面无明显错位
```

### 面试可讲点

```text
为什么后台接口和前台接口要分开
为什么前台接口必须过滤 status = 1
文章和标签为什么是多对多关系
文章状态流转如何设计
DTO、VO、Entity 为什么要分层
```

### 当前需要处理

```text
不要提交运行日志 blog/logs/*
补充 README 项目启动说明
补充接口文档或截图
准备 5-10 条演示文章数据
```

## 六、阶段 6：MinIO 文件上传系统

### 为什么优先做

博客项目如果只能手动填写图片 URL，工程完整度不够。文件上传是后端面试高频工程题，也非常适合用来展示对象存储、文件校验、URL 存储等能力。

### 目标

实现后台图片上传能力，优先服务：

```text
文章封面图
Markdown 正文插图
合集封面
后续网站 Logo / 头像
```

### 推荐实现范围

后端新增：

```text
MinIO 配置
FileController
FileService
FileVO
文件类型校验
文件大小限制
文件名防重复
上传后返回 URL
```

接口建议：

```text
POST /admin/file/upload
```

返回示例：

```json
{
  "code": 200,
  "message": "success",
  "data": {
    "url": "http://localhost:9000/kyonelife/article/2026/06/xxx.jpg",
    "filename": "xxx.jpg"
  }
}
```

前端改造：

```text
文章编辑页封面图支持上传
Markdown 编辑器支持上传图片并插入图片链接
合集封面支持上传
```

### 数据存储原则

```text
图片文件本体存 MinIO
图片 URL 存 MySQL
文章内容仍存 Markdown
```

不要把图片二进制存进 MySQL。

### 面试可讲点

```text
为什么使用对象存储而不是把文件存在数据库
文件名如何防止重复
如何限制文件类型和大小
如何避免用户上传危险文件
为什么 URL 存数据库，文件存 MinIO
MinIO 后续如何迁移到阿里云 OSS / 腾讯云 COS / S3
```

### 验收标准

```text
后台文章封面可以上传并回显
Markdown 正文可以插入上传图片
前台文章详情可以正常展示图片
非法文件类型会被拒绝
超大文件会被拒绝
MinIO 中能看到上传文件
```

## 七、阶段 7：后台操作日志系统

### 为什么要做

操作日志是典型工程题。它能体现：

```text
AOP
拦截器
异步处理
异常记录
审计意识
后台查询
```

比单纯做一个页面更有工程含量。

### 目标

自动记录管理员在后台的关键操作。

记录内容：

```text
操作人 ID
操作人名称
模块
操作类型
请求方法
请求路径
请求参数
响应状态
IP
User-Agent
耗时
异常信息
创建时间
```

### 推荐实现

数据库表：

```text
operation_log
```

后端：

```text
自定义注解 @OperationLog
AOP 切面记录日志
异步线程池写入数据库
日志分页查询接口
日志详情接口
按时间清理接口
```

接口建议：

```text
GET /admin/operation-log/page
GET /admin/operation-log/{id}
DELETE /admin/operation-log/clean?before=2026-01-01
```

后台页面：

```text
操作日志列表
按操作人筛选
按模块筛选
按时间筛选
查看请求参数和异常详情
```

### 面试可讲点

```text
为什么用 AOP 记录，而不是每个 Controller 手写
为什么日志写入要异步
异步写入失败如何处理
如何避免记录敏感字段，比如密码
日志量变大后如何演进到 MQ + Elasticsearch / ClickHouse
```

### 验收标准

```text
新增文章会记录日志
编辑文章会记录日志
删除文章会记录日志
登录失败或接口异常能记录异常信息
后台可以分页查询日志
密码等敏感字段不会明文记录
```

## 十三、简历写法建议

不要写：

```text
完成个人博客系统，实现文章管理、标签管理、评论管理等功能。
```

这样太普通。

建议写：

```text
基于 Spring Boot + Vue3 实现个人内容管理与展示平台，采用前后台接口隔离设计，后台支持文章草稿、发布、下架、标签/合集管理，前台仅暴露已发布内容，并通过 DTO/VO/Entity 分层避免内部字段泄漏。
```

后续做完 MinIO 和日志后，可以写：

```text
接入 MinIO 对象存储实现文章封面和 Markdown 图片上传，设计文件类型校验、文件名防重复和 URL 持久化方案，避免大文件直接进入数据库。
```

做完操作日志后，可以写：

```text
基于自定义注解和 Spring AOP 实现后台操作日志，自动记录管理员、请求路径、参数、耗时、异常信息，并通过异步写入降低对业务接口响应时间的影响。
```

做完缓存后，可以写：

```text
针对前台文章详情、标签列表等高频读接口引入 Redis 缓存，并在文章更新后主动失效缓存，降低数据库查询压力。
```

## 十四、面试回答框架

遇到“你这个项目有什么亮点”时，不要只说功能。建议按这个顺序说：

```text
1. 项目目标：个人内容管理与展示平台
2. 核心链路：后台写作发布，前台展示阅读
3. 工程设计：前后台接口隔离、统一异常、鉴权、跨域、DTO/VO 分层
4. 重点模块：文件上传、操作日志、搜索、缓存
5. 演进思路：MVP 先跑通，数据量上来后引入 MQ、搜索引擎、对象存储、缓存
```

遇到“如果让你设计一个日志系统”时，可以结合项目回答：

```text
我会先按后台操作日志设计。通过自定义注解标记需要记录的接口，用 AOP 在方法执行前后记录请求路径、参数、管理员、耗时和异常信息。MVP 阶段可以异步写 MySQL，方便后台分页查询；如果日志量变大，可以引入 MQ 解耦，再写入 Elasticsearch 或 ClickHouse 做检索分析。
```
