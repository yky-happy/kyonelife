# kyonelife 下一步开发计划

## 一、当前项目判断

项目当前已经过了“搭架子”阶段，进入了“跑通核心业务链路”的阶段。

从代码和文档看，当前完成度如下：

| 模块 | 当前状态 | 说明 |
| --- | --- | --- |
| 后端基础设施 | 已完成 | Spring Boot、统一响应、异常处理、MyBatis Plus、Sa-Token、Knife4j、CORS 等基础能力已落地 |
| 数据库结构 | 已完成主体设计 | 已有 14 张核心表，覆盖用户、管理员、角色、菜单、文章、标签、合集、评论、轮播、配置等 |
| 管理员登录 | 已完成基础链路 | 后端登录接口和前端登录页/Token 拦截逻辑已具备 |
| 标签管理 | 已完成端到端 CRUD | 后端接口、前端 API、列表、新增、编辑、删除、分页搜索均已实现 |
| 合集管理 | 已完成端到端 CRUD | 与标签管理类似，已具备完整 CRUD |
| 文章管理 | 只有前端列表壳 | 页面存在，但仍是静态结构；没有后端 ArticleController、ArticleService 和前端 article API |
| 写文章页面 | 尚未实现 | 文章列表里跳转 `/content/article/edit`，但路由和页面还没有配置 |
| 前台 blog-web | 只有骨架 | 还未进入文章列表、详情、合集、标签等展示功能 |
| 评论、轮播、配置、系统管理 | 页面骨架为主 | 适合等文章主链路完成后再推进 |

因此，下一步不建议继续横向铺很多管理模块，而应该优先做“文章管理”这个主链路。

## 二、下一阶段目标

下一阶段建议命名为：

**阶段 4：文章管理 MVP**

目标是先跑通后台文章从创建到展示列表的完整流程：

1. 管理端可以新建文章。
2. 管理端可以编辑文章。
3. 管理端可以查询文章列表。
4. 管理端可以删除文章。
5. 管理端可以发布、下架、保存草稿。
6. 文章可以选择标签和合集。
7. 文章正文先支持 Markdown 原文保存，HTML 渲染内容可以同步保存或后续补强。

完成这个阶段后，项目才真正具备个人博客的核心生产能力。

## 三、为什么下一步优先做文章管理

文章是整个博客系统的中心数据。

标签和合集已经完成，但它们本质上是文章的辅助数据。如果不做文章管理，前台展示、评论、搜索、AI 摘要、阅读量、归档页都没有真实业务数据可以承接。

从依赖关系看，合理顺序是：

```text
登录
  -> 标签 CRUD
  -> 合集 CRUD
  -> 文章 CRUD
  -> 前台文章列表/详情
  -> 评论/轮播/配置/AI/搜索
```

当前已经完成到“合集 CRUD”，所以最自然的下一步就是文章 CRUD。

## 四、文章管理 MVP 范围

### 4.1 本阶段要做

本阶段只做文章最小可用能力：

- 文章分页列表
- 按标题搜索
- 按状态筛选
- 新增文章
- 编辑文章
- 删除文章
- 发布/下架
- 保存草稿
- 选择合集
- 选择多个标签
- 设置封面图地址
- 设置摘要
- 设置原创/转载
- 设置 SEO 关键词
- 设置置顶

### 4.2 本阶段暂不做

以下能力建议暂缓，避免文章主链路变重：

- 图片上传
- AI 摘要
- AI 标签推荐
- 阅读量 Redis 异步刷库
- 文章内容图片独立入库
- 轮播文章最多 5 篇的完整校验
- 动态权限按钮控制
- 前台文章详情渲染优化

这些都可以在文章 CRUD 跑通后单独迭代。

## 五、后端开发计划

### 5.1 新增 DTO

建议新增以下文件：

```text
blog/src/main/java/com/yky/blog/admin/dto/ArticleSaveDTO.java
blog/src/main/java/com/yky/blog/admin/dto/ArticleStatusDTO.java
```

`ArticleSaveDTO` 建议字段：

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| title | String | 是 | 文章标题 |
| cover | String | 否 | 封面图地址 |
| summary | String | 否 | 摘要 |
| content | String | 否 | HTML 内容，MVP 可先等于 Markdown 渲染后内容或暂存空 |
| contentMd | String | 是 | Markdown 原文 |
| keywords | String | 否 | SEO 关键词 |
| aiDescribe | String | 否 | AI 简短描述，MVP 可不填 |
| collectionId | Long | 否 | 合集 ID |
| tagIds | List<Long> | 否 | 标签 ID 列表 |
| status | Integer | 是 | 0 草稿，1 已发布，2 已下架 |
| isStick | Integer | 否 | 是否置顶 |
| isCarousel | Integer | 否 | 是否加入轮播 |
| carouselSort | Integer | 否 | 轮播排序 |
| isOriginal | Integer | 是 | 是否原创 |
| originalUrl | String | 否 | 原文链接 |

`ArticleStatusDTO` 建议字段：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| status | Integer | 0 草稿，1 已发布，2 已下架 |

### 5.2 新增 VO

建议新增：

```text
blog/src/main/java/com/yky/blog/admin/vo/ArticleVO.java
blog/src/main/java/com/yky/blog/admin/vo/ArticleDetailVO.java
```

`ArticleVO` 用于列表，字段尽量轻：

- id
- title
- cover
- summary
- collectionId
- collectionName
- tags
- status
- isStick
- isCarousel
- viewCount
- createTime
- updateTime

`ArticleDetailVO` 用于编辑回显，可以包含完整正文：

- ArticleVO 的基础字段
- content
- contentMd
- keywords
- aiDescribe
- originalUrl
- carouselSort
- isOriginal
- tagIds

### 5.3 新增 Service

建议新增：

```text
blog/src/main/java/com/yky/blog/admin/service/ArticleService.java
blog/src/main/java/com/yky/blog/admin/service/impl/ArticleServiceImpl.java
```

核心方法：

```java
IPage<ArticleVO> pageArticle(int page, int size, String keyword, Integer status);
ArticleDetailVO getArticleDetail(Long id);
Long saveArticle(ArticleSaveDTO dto);
void updateArticle(Long id, ArticleSaveDTO dto);
void updateArticleStatus(Long id, Integer status);
void removeArticle(Long id);
```

实现重点：

1. 保存文章主表。
2. 保存文章与标签的关联表 `article_tag`。
3. 编辑文章时先删除旧标签关联，再插入新标签关联。
4. 删除文章时同步删除 `article_tag` 关联。
5. 查询列表时补齐合集名称和标签列表。
6. 不要把复杂 SQL 一次写满，MVP 可以先用多次简单查询组装 VO。

### 5.4 新增 Controller

建议新增：

```text
blog/src/main/java/com/yky/blog/admin/controller/ArticleController.java
```

接口建议：

| 方法 | 路径 | 用途 |
| --- | --- | --- |
| GET | `/admin/article/page` | 分页查询文章 |
| GET | `/admin/article/{id}` | 查询文章详情 |
| POST | `/admin/article` | 新增文章 |
| PUT | `/admin/article/{id}` | 编辑文章 |
| PATCH | `/admin/article/{id}/status` | 发布/下架/草稿 |
| DELETE | `/admin/article/{id}` | 删除文章 |

### 5.5 后端验收标准

后端完成后，必须能在 Knife4j 里完成以下自测：

1. 新增一篇草稿文章。
2. 查询文章分页列表，能看到刚新增的文章。
3. 查询文章详情，能回显正文、合集、标签。
4. 编辑标题、摘要、标签后，再查详情能看到变化。
5. 发布文章，列表状态变为已发布。
6. 下架文章，列表状态变为已下架。
7. 删除文章后，列表不再出现，`article_tag` 关联也被清理。

### 6.5 Markdown 编辑器选择

当前阶段建议选 `md-editor-v3`，理由：

- Vue 3 集成简单。
- 足够支撑个人博客写作。
- 比 vditor 更轻，前期配置成本低。
- 后续如果需要更强的所见即所得体验，再替换也不晚。

本阶段可以先保存 `contentMd`，`content` 字段可以先存编辑器产出的 HTML，或暂时与 Markdown 渲染结果保持一致。

8. 编辑草稿并发布。
9. 列表状态变为已发布。
10. 下架文章。
11. 删除文章。

## 八、风险点和注意事项

### 8.1 数据库脚本存在库名不一致

`init.sql` 顶部有：

```sql
CREATE DATABASE IF NOT EXISTS kyonelife ...
USE kyonelife_blog;
```

这里数据库名不一致。后续继续开发前建议先统一，否则新环境初始化时容易失败。

建议统一为 `kyonelife`，并检查 `application.yml` 的数据库连接名是否一致。

### 8.2 文章标签关联不要漏

文章和标签是多对多关系，保存文章时不能只保存 `article` 表，还要处理 `article_tag`。

编辑文章时建议采用简单可靠的方案：

1. 删除该文章旧的 `article_tag`。
2. 根据新的 `tagIds` 批量插入。

### 8.3 先别追求复杂 SQL

文章列表需要合集名、标签列表、文章数等信息，容易诱导一开始就写复杂 JOIN。

MVP 建议：

1. 分页查文章主表。
2. 批量查合集。
3. 批量查文章标签关联。
4. 批量查标签。
5. 在 Java 里组装 VO。

这样可读性更高，也适合当前学习阶段。

### 8.4 删除文章要同步清理关联

删除文章时至少要清理：

- `article`
- `article_tag`

`article_image` 当前阶段如果还没使用，可以后续再处理。但一旦文章图片入库，就要同步设计删除策略。

### 8.5 先保留静态权限菜单

目前可以继续使用前端静态路由和侧边栏。动态菜单、按钮权限属于系统管理阶段，不要阻塞文章主链路。

## 九、完成阶段 4 后的下一步

文章管理 MVP 完成后，建议进入：

**阶段 5：前台展示 MVP**

优先做：

1. 前台首页文章列表。
2. 文章详情页。
3. 标签筛选文章。
4. 合集详情页。
5. 归档页。

这样后台能写文章，前台能看到文章，个人博客的最小闭环就真正成立。

评论、轮播、网站配置、AI 摘要、图片上传都可以在这个闭环之后逐步补。
